Use case

Run repeatable checks on public metadata, catalog exports, and harvest workflows.

Solvyr supports reviewable batch runs over public or test metadata before wider publication, harvesting, indexing, or downstream reuse. It is not a catalog replacement; it is a controlled run that returns issue lists, suggested fixes, caveats, and a decision-ready report.

Scope a public-data run See the proof run

Catalog quality

Find issues before records move downstream.

Check records for missing fields, broken links, stale references, inconsistent classifications, duplicates, and profile drift.

Recurring runs

Repeat the same workflow after exports refresh.

Useful when metadata, portals, indexes, or reporting systems are refreshed on a predictable cycle.

Review package

Return visible evidence, not silent pass/fail.

Each run should produce issue data, record-level notes, summary counts, caveats, and recommended next actions.

Use this when

  • You manage a public-data, archive, library, or research-data catalog.
  • You exchange metadata through DCAT, XML, CSV, JSON, or portal exports.
  • You need checks before publication, harvesting, indexing, or reuse.
  • The data is public, low-sensitivity, redacted, or safe test data.
  • An internal owner can review suggested fixes before anything changes.

Not for

  • Replacing CKAN, GeoNetwork, Dataverse, or other catalog software.
  • Automatic production writes without human review.
  • Compliance certification or formal data-quality accreditation.
  • Highly sensitive data without prior agreement.
  • Realtime monitoring or incident response.

Output

What a public-data quality run returns.

A typical bundle can include issue CSV/JSON, record-level notes, broken-link and stale-reference findings, schema or profile warnings, duplicate and classification signals, enrichment suggestions, summary counts, caveats, and a reviewed recommendation for the next run.

Boundary

Start with public or test metadata and a narrow acceptance rule.

This is strongest when the first run has a bounded export, visible validation criteria, and a clear reviewer. Solvyr should not receive sensitive production data through public forms or email; transfer, retention, deletion, and operator access need a written scope first.
