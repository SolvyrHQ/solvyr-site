Fit

For teams that need a real corpus, not another vague data project.

Fast Corpus fits when the input is clean enough, the output is useful as JSONL, and the workload can be handled asynchronously.

Good fit

  • Thousands of clean digital PDF pages or text files
  • RAG, search, evals, fine-tuning prep, or agent memory
  • Embedding refreshes, re-indexing, enrichment, or backfills
  • A representative non-sensitive sample can be shared

Not the current fit

  • Realtime chat or model serving
  • OCR-heavy scans unless scoped separately
  • Custom structured extraction in the current trial
  • High-sensitivity data without prior agreement

Strong buyer signals

  • A backlog of documents waiting to be made usable
  • A recurring index or corpus refresh
  • A team already building RAG, search, eval, or agent systems
  • A clear owner for input quality and output validation

Common paths

  • Reprocessing an archive into searchable chunks
  • Preparing a RAG corpus from PDFs or text exports
  • Building an eval or fine-tuning prep dataset
  • Creating a repeatable corpus refresh process

Model

Fast Corpus is the wedge. Controlled execution is the system underneath.

The product story should start with the corpus result. The deeper Solvyr system matters because it can make async work observable, bounded, and economically useful.

Completed work, not raw GPU rental

Customers should buy usable corpus output and run evidence, not abstract GPU hours.

Async workloads fit the operating model

Backlogs, refreshes, enrichment, and reprocessing can tolerate queueing better than realtime product paths.

Recurring refresh is the business layer

Security

Security for a bounded pilot: lifecycle, access, deletion, and data minimization.

Practical hygiene for low- or moderate-sensitivity pilot workloads, with clear limits before data runs.

What we can define for a pilot

  • Data lifecycle, retention, and deletion
  • Input and output storage locations
  • Scoped operator access
  • Data minimization before execution
  • Payload logging checks
  • Temporary scratch cleanup checks

Current boundaries

  • No high-sensitivity workloads without a safe pilot dataset
  • No cryptographic privacy or confidential-compute claims
  • No broad enterprise compliance claim yet
  • No claim that distribution itself creates confidentiality

Risk reduction

Redaction or pseudonymization can be part of the pilot design.

We can help define pre-upload redaction or pseudonymization. Keep rehydration customer-side. Risk reduction, not guaranteed anonymity.

Practical promise

Security starts with workload selection and honest boundaries.

Some workloads stay out of scope. Suitable pilots get lifecycle, access, deletion, and execution boundaries before data is shared.

Security appendix

Topics to settle before a pilot run.

Data lifecycle
What is uploaded, produced, retained, deleted, and when.
Scoped access
Who can see payloads, logs, outputs, and operational metadata.
Redaction
Which fields can be removed, replaced, or kept customer-side.
Logging and scratch
What payload logging is allowed and how temporary files are cleaned.
